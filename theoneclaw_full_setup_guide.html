TheOneClaw — Full Setup Guide

A single top-to-bottom checklist for a $0-first AI assistant stack using OpenRouter free models, Orgo computers, OpenClaw, Telegram, Paperclip, and optional Paperless-ngx.

What this guide is

This guide is built to be followed from top to bottom with no guessing. The first goal is one working AI loop. After that, each tool is added only when the previous step is visibly working.

Verified rule: OpenRouter’s openrouter/free router is free and routes to free models; OpenClaw supports Telegram and iMessage channels; Orgo’s default computer spec is 4GB RAM / 2 cores; Paperclip starts locally with npx paperclipai onboard --yes; Paperless-ngx has documented setup and Docker-based install options.

Simple architecture

User
  ↓
OpenRouter free model
  ↓
OpenClaw Gateway
  ↙            ↘
Telegram        Orgo cloud computer
  ↓                 ↓
Paperclip control plane  →  Paperless-ngx archive (optional)

What you need before starting

Do not start with Notion, screen recording learning, hotkeys, or self-play until the base loop is working. Those are later layers.

PHASE 1 — Get free AI working

Goal: send one prompt to OpenRouter and get one answer back.

1) Make the OpenRouter account

2) Test the free model router

export OPENROUTER_API_KEY="your_key_here"

curl https://openrouter.ai/api/v1/chat/completions   -H "Authorization: Bearer $OPENROUTER_API_KEY"   -H "Content-Type: application/json"   -d '{
    "model": "openrouter/free",
    "messages": [
      {"role": "user", "content": "Explain binary search in one paragraph."}
    ]
  }'
Success looks like: a JSON response with an answer in choices[0].message.content.
Most likely error: bad API key or missing environment variable. Fix: re-copy the key, re-run export OPENROUTER_API_KEY=..., and test again.

Why this works: OpenRouter says the free router is free, and the :free variant is the cheapest way to try free models. verified

PHASE 2 — Set up Orgo as the sandbox computer

Goal: create a cloud computer for tasks that need a real desktop, browser, screenshots, clicks, or file actions.

1) Create a workspace

curl -X POST https://www.orgo.ai/api/workspaces   -H "Authorization: Bearer $ORGO_API_KEY"   -H "Content-Type: application/json"   -d '{"name": "theoneclaw"}'

2) Create a 4GB computer

curl -X POST https://www.orgo.ai/api/computers   -H "Authorization: Bearer $ORGO_API_KEY"   -H "Content-Type: application/json"   -d '{
    "workspace_id": "YOUR_WORKSPACE_ID",
    "name": "sandbox-1",
    "os": "linux",
    "ram": 4,
    "cpu": 2
  }'

3) Use the computer

# Take a screenshot
curl https://www.orgo.ai/api/computers/YOUR_COMPUTER_ID/screenshot   -H "Authorization: Bearer $ORGO_API_KEY"

# Click
curl -X POST https://www.orgo.ai/api/computers/YOUR_COMPUTER_ID/click   -H "Authorization: Bearer $ORGO_API_KEY"   -H "Content-Type: application/json"   -d '{"x":100,"y":200}'
Success looks like: the computer exists, responds, and can be controlled.
Free-tier computers auto-stop after 15 minutes of inactivity. Use that to keep costs low and clean up when done.
Most likely error: invalid workspace or computer ID. Fix: create the workspace first, copy the returned ID, then create the computer.

PHASE 3 — Install OpenClaw and run the Gateway

Goal: make OpenClaw the bridge between your chat apps and your agent runtime.

1) Install and onboard

openclaw onboard

If you are building manually, OpenClaw’s docs say onboarding is the fastest path. OpenClaw also has a status/doctor flow for diagnostics.

2) Verify the Gateway

openclaw status
openclaw status --all
openclaw doctor
Success looks like: the gateway is reachable and the doctor report is clean or at least clearly tells you what to fix.
Most likely error: the gateway is not running or the config is missing. Fix: run openclaw onboard again, then openclaw status.

3) Create the base config idea

{
  "agent": { "workspace": "~/.openclaw/workspace" },
  "channels": {
    "telegram": {
      "botToken": "YOUR_TELEGRAM_BOT_TOKEN",
      "allowFrom": ["YOUR_TELEGRAM_USER_ID"],
      "dmPolicy": "pairing"
    }
  }
}

OpenClaw’s docs show Telegram as the fastest chat channel, and DM pairing/allowlists are the recommended safety model. verified

PHASE 4 — Connect Telegram

Goal: message your agent from your phone.

1) Make the bot

2) Add the bot token to OpenClaw

export TELEGRAM_BOT_TOKEN="your_bot_token"

OpenClaw’s docs say the token can live in channels.telegram.botToken or fallback to TELEGRAM_BOT_TOKEN for the default account.

3) Approve your own DM

openclaw pairing list telegram
openclaw pairing approve telegram YOUR_CODE
Success looks like: you send the bot a DM and OpenClaw replies after the pairing approval.
Most likely error: privacy mode or allowlist issue. Fix: approve pairing, confirm your numeric Telegram user ID, and keep DM policy strict.

Telegram supports long polling and webhooks, but OpenClaw’s docs make Telegram the fastest setup path. verified

PHASE 5 — Add Paperclip as the control plane

Goal: use Paperclip for company structure, goals, approvals, budgets, and task ownership.

1) Install Paperclip locally

npx paperclipai onboard --yes

Paperclip’s local deployment uses embedded PostgreSQL and starts on http://localhost:3100.

2) Create your company

3) Hire the CEO agent

4) Approve the CEO strategy

Paperclip says the first CEO strategy must be approved before it can keep delegating work.

Success looks like: the company exists, the CEO is hired, and the first strategy is approved.
Most likely error: missing Node.js or pnpm. Fix: install Node 20+ and pnpm 9.15+, then run onboarding again.

PHASE 6 — Make Paperclip talk to OpenClaw

Goal: let Paperclip assign work to OpenClaw through an adapter.

Recommended path

{
  "adapterType": "http",
  "adapterConfig": {
    "url": "https://YOUR-OPENCLAW-ENDPOINT/invoke",
    "method": "POST",
    "headers": {
      "Authorization": "Bearer YOUR_OPENCLAW_TOKEN"
    },
    "timeoutMs": 30000,
    "payloadTemplate": {
      "agentId": "{{agent.id}}",
      "companyId": "{{company.id}}",
      "runId": "{{run.id}}"
    }
  }
}

Paperclip’s adapter guide says HTTP adapters are for remote agent execution and OpenClaw integration. OpenClaw docs also support webhook security and pairing.

If you cannot verify your exact OpenClaw endpoint, stop here and use the local OpenClaw gateway first. Do not invent a URL.
Success looks like: Paperclip creates a task, OpenClaw receives it, and the reply comes back into the Paperclip run log.

PHASE 7 — Turn this into useful daily workflows

Now the system can help with the kinds of work you actually want:

  • Study prompts, exam revision, flashcards, and summaries
  • Vibecoding tasks: app ideas, small features, game loops, project planning
  • Research outlines and quick reports
  • IT tasks: troubleshoot, explain logs, summarize docs, suggest fixes
  • Creative work: song ideas, image prompts, video outlines, diagrams
Use skills folders in OpenClaw when a repeatable workflow becomes useful. OpenClaw’s skills system uses SKILL.md files and supports workspace-level and agent-level skill storage.

PHASE 8 — Optional iMessage and shared inbox paths

iMessage

OpenClaw supports iMessage. The docs say the recommended path is BlueBubbles, and the imessage channel exists too. The iMessage path requires Full Disk Access for the Messages database if you use the direct iMessage route.

{
  "channels": {
    "bluebubbles": {
      "enabled": true,
      "dmPolicy": "pairing"
    }
  }
}
This is an optional later phase. Add it only after Telegram is stable.

Shared mailbox / Gmail

OpenClaw also has Gmail Pub/Sub automation docs. That path uses Gmail watch → Pub/Sub → OpenClaw webhook, so it is an advanced email bridge rather than the first thing to build.

PHASE 9 — Optional Paperless-ngx archive

Goal: store processed documents in a searchable archive.

Docker-style setup

git clone https://github.com/paperless-ngx/paperless-ngx
cd paperless-ngx
docker compose up -d

Paperless-ngx’s docs say it is a community-supported document management system with setup, configuration, troubleshooting, and Docker-based installation options.

Success looks like: the archive opens, documents can be stored, searched, and retrieved.
Most likely error: Docker not installed. Fix: stop here and keep Paperless-ngx for later.

Weekly self-review

Use this each week once the system is running:

  • What tasks were completed?
  • What got stuck?
  • What took too many tokens or too much time?
  • Which new skill should be saved into an OpenClaw skill folder?
  • What should be automated next?

First success test

/ask explain loops like I am learning Python

Expected result: the bot replies with a short, clear explanation and does not ask you to configure anything else first.

Troubleshooting

  • OpenRouter key fails: re-run the export command and check the header.
  • OpenRouter free router errors: try again later or use a different :free model.
  • Orgo computer disappears: free-tier computers auto-stop after inactivity; create it again.
  • OpenClaw not replying: run openclaw status, then openclaw doctor.
  • Telegram bot silent: confirm pairing approval and allowlist user ID.
  • Paperclip won’t start: check Node.js 20+ and pnpm 9.15+.
  • Paperless-ngx install fails: stop and only use it after the core system works.

What is intentionally left as later work

Screen recording-to-learning, hotkey-triggered workflows, self-play loops, and Notion automation are powerful ideas, but I left them as later phases because the verified docs here support a safer, smaller build first.
